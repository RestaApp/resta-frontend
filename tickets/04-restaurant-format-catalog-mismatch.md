# Тикет 4 — Расхождение каталога `restaurant_formats` и данных

**Тип:** data / config
**Приоритет:** низкий

## Проблема
В реальных данных у заведений встречаются значения `restaurant_format`, которых **нет** в каталоге `GET /api/v1/catalogs/restaurant_formats`.

**Каталог (13 значений):** `full_service, cafe, bistro, fast_food, sushi_bar, bar, pizzeria, bakery, food_truck, catering, gastromarket, club, canteen`.

**В данных дополнительно встречаются (5 значений вне каталога):** `restaurant`, `steakhouse`, `fine_dining`, `pub`, `buffet`.

Проверено через `GET /api/v1/users?user_type=restaurant` под поставщиком — форматы заведений в списке содержат эти значения.

## Последствие
Любой клиент, который маппит формат по каталогу, для этих 5 значений не найдёт соответствия. На фронте это вызывало смешанный вид списка (часть форматов локализована, часть — сырым текстом). **На фронте уже закрыто** добавлением лейблов для этих значений (`labels.restaurantFormat.*` в ru/en), но первоисточник расхождения — на бэке.

## Что сделать (один из вариантов)
1. **Расширить каталог** `restaurant_formats`, добавив `restaurant`, `steakhouse`, `fine_dining`, `pub`, `buffet` (если это валидные форматы) — тогда они станут официальными и доступными в формах выбора.
2. **Либо** нормализовать данные: смигрировать существующие записи на ближайшие каталожные значения (напр. `restaurant → full_service`, `pub → bar`, `buffet → catering`/`canteen`, `steakhouse`/`fine_dining → full_service`) и закрыть запись формата валидацией по каталогу, чтобы новые значения вне каталога не появлялись.

## Критерий готовности
- Множество значений `restaurant_format` в данных ⊆ каталога `restaurant_formats`.
- Создание/редактирование заведения не допускает формат вне каталога (валидация по каталогу).

## Контекст
Не блокирует пользователя; фронт отображает корректно за счёт фолбэк-лейблов. Задача — про целостность каталога и предсказуемость данных.
